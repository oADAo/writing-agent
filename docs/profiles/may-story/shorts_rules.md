# 玫玫物語 Shorts 文本規則

## 用途

這份文件只負責一件事：把遊戲相關題目寫成適合 `YouTube Shorts / 抖音型短影音` 的口播文本。

它不是長影片縮短版，也不是精準剪輯時間軸。
它的任務是快速打穿一個最值得講的 punch，並把文案先拆成固定的字幕版型分類。

## 目前固定預設

- 只要使用者說要做 `Shorts`，預設就照目前這套結果格式和聲線輸出。
- 這套預設要用通用型方式套用到不同遊戲、不同題型；不要把某支稿的專有名詞、場景或梗直接硬套到別的主題。
- 預設交付固定是：
  - `Hook Title`
  - `Hook Burst Text`
  - `Template Marked Script`
- `Template Marked Script` 只保留 `[hook]`、`[intro]`、`[chapter]`
- 最後收尾句直接接在文稿最後，不另外標 `[ending]`
- 如果之後使用者再貼親自潤過的版本，以最新親改版優先覆蓋這套預設。

## 適用題型

Shorts 可以寫：
- 劇情悲劇型
- 冷知識型
- 隱藏設定型
- 反轉型
- 提醒型
- 攻略型
- `千萬別做什麼` 型

但一支短稿只准有一個主 punch，不要一支塞三個重點。

## 核心寫法

- 第一行就要鉤人。
- 第二段就要點出核心事件或反差。
- 中段要一路往前推，不要原地打轉。
- 最後一句要收在重擊、荒謬、心痛、反轉或震撼點。
- 如果使用者有指定參考 Shorts，就要真的拆那支 Shorts 的節奏、語氣、資訊密度，再重寫，不准只拿主題換句話說。
- 正式成品不能只交一整篇稿，要先把文案拆進固定的字幕版型。
- 如果使用者沒有指定其他結構，預設先用 `Hook -> Intro -> 3 塊重點 -> 收尾` 這個通用模板，再壓成 Shorts。

Shorts 最常見的節奏：

1. 直接鉤住人
2. 快速點核心事件
3. 用最少背景把事情推到最痛的地方
4. 最後一拳打完

## 字幕版型分類

每支 Shorts 預設都先拆成下面四類：

### 1. `HOOK 爆字`

- 用在第一拳。
- 要短、狠、有衝擊感。
- 預設維持單行。
- 寫法像：
  - `這個先別做`
  - `家具可以搬回家`
  - `這招很多人不知道`
- 視覺目標：
  - 紅色爆字
  - 有衝擊感
  - 配全螢幕白色速度線

### 2. `前言穩字`

- 用在前兩句之後，補主 promise 或補一句「為什麼這件事重要」。
- 可以比爆字完整，但仍然要短。
- 主要是穩住觀眾，不是開始長篇解釋。
- 視覺目標：
  - 中間黑色半透明底版
  - 白字
  - 一次放 `1 到 3` 句短文字

### 3. `步驟提示卡`

- 用在節奏切點或教學步驟切換。
- 只放很短的步驟主題，不要寫成長影片章節名。
- 一張卡只打一個很短的步驟。
- 視覺目標：
  - 全螢幕
  - 中間大塊黑色半透明背板
  - 中間白色大字放步驟主題
  - 右下角大數字顯示第幾步

### 4. `教學穩字`

- 這是最常出現的正文字幕。
- 主要任務是把操作、條件、結果穩定交代清楚。
- 預設永遠維持單行字量。
- 如果一句超過單行安全長度，就重寫或拆成下一張，不要硬塞成兩行。
- 視覺目標：
  - 中間偏下位置
  - 有簡單框線或底框
  - 不要太多花俏特效

## 版型使用原則

- `HOOK 爆字` 負責第一拳，不拿來講完整資訊。
- `前言穩字` 負責補 promise，不拿來塞完整教學。
- `步驟提示卡` 負責切節奏，不拿來寫長句。
- `教學穩字` 才是正文主體，而且要盡量維持單行。
- 正式文稿裡也要把模板標記直接寫出來，至少要能看見：
  - `[hook]`
  - `[intro]`
  - `[chapter] 第一步 ...`
  - `[chapter] 第二步 ...`
  - `[chapter] 第三步 ...`
- 最後收尾那一句直接接在文稿最後，不要另外標 `[ending]`
- 如果這支是 `教學 / 建造 / 配置` 類型，三個 `[chapter]` 要真的對應三個明確步驟，不能只是在介紹概念。
- `[chapter]` 後面不要寫 `第一章` 這種不能直接念的標籤。
- 要根據文稿意思改成 `第一步`、`第二步`、`第一個`、`第二個重點` 這種自然可唸的寫法。
- 如果要標後製重點字，預設用 `「」` 標記。
- 重點字可以標在 `intro`、章節正文和最後收尾那一句，不要標 `hook`。
- 也不要標 `[chapter]` 那一行本身，像 `第一步`、`第二步` 這些步驟標籤不要上色。
- 名詞標記量可以比以前更高，後面章節也要持續有 `「」`，不要只集中在前面。
- 但還是要以好讀為前提，不要整句都標，也不要每個字都標。
- 如果參考 Shorts 本身有很強的版型習慣，正式改寫時也要把版型層級一起拆出來，不准只交純文字。

## 聲線原則

- 保留 `玫玫物語` 的成熟、親切、講人話。
- 但比長影片更直接、更狠、更情緒。
- 不是慢慢教，而是像一口氣把最重點的東西打進觀眾腦袋。
- 可以有情緒張力，但不能只剩喊情緒。
- 如果使用者直接貼自己的潤稿版本，優先把那版當成最新聲線樣本；下一次同題型先照那個句長、轉場和收尾感起稿。
- 在講 `有趣、荒謬、穿幫、反差` 這種 punch 時，不要用 `不是...而是...` 當主軸句型。
- 這種句型太像在修正觀念，不像在帶觀眾看有趣畫面。

## 長度原則

- 預設以約 `300 到 400` 字中文口播稿為主。
- 句子可以長短混用，但要順口。
- 念起來要像人會一口氣講完的，不要像條列筆記。
- `intro` 要再壓短，不要一開頭就寫成一大段前言。

## 資料使用原則

- Shorts 也不能亂寫，尤其是劇情、設定、攻略、隱藏條件這類內容。
- 如果資訊會受版本、翻譯、隱藏設定或社群誤讀影響，先重新查。
- 只保留真的能推進 punch 的背景資訊，多餘背景要砍掉。

## 主題來源原則

- Shorts 題型優先直接找 `真正有流量的 Shorts` 來抄主題。
- `YouTube` 只認 `/shorts/` 連結；不要把一般影片、切片或只是秒數短的片混進來。
- 只看真的跑量的 Shorts，不要把普通短片或只是秒數短的影片硬算進來。
- Shorts 題型不能只靠單支低觀看 Shorts 或偶發切片硬抄。
- 如果這款遊戲本來就缺強勢 Shorts 生態，才回到 `模式 1：熱門主題搜尋` 的跨語圈驗證結果。
- `有趣一幕`、`臨場反應`、`事故切片` 只有在其他語圈或其他創作者也反覆出現時，才算可複製題型。
- 主證據優先看 `小型自媒體 / 個人創作者 / 非官方搬運解析`，官方與大媒體只留作事實校正。
- 找參考片之前，先做遊戲名翻譯表：`中文名 / 日文名 / 英文名`；有常見簡稱、舊譯名、玩家暱稱也一起列。
- 各語圈要用自己的原生關鍵字分開搜，而且 `一題一搜`，不要一口氣混很多問題。
- 先拉一批有量候選，再分成 `主題簇`，不要被單支爆片牽著走。
- 常見可整理的主題簇包含：
  - `實測會發生什麼`
  - `離譜互動 / 彩蛋`
  - `建築 / 畫面奇觀`
  - `高收益技巧`
  - `新手別犯錯`
  - `稀有 / 傳說 / 隱藏解鎖`
  - `劇情反轉 / 角色設定`
  - `Bug / 荒謬現象 / 搞笑事故`
- 只留下 `可複製熱門`：
  - 不同創作者都做過
  - 不同語言圈也有人做
  - 標題一看就懂
  - 畫面一眼就懂
  - 不靠創作者本人也成立
- 之後評估時固定問這 `5 個問題`：
  - 這題 `3 秒內` 看得懂嗎
  - 有明確結果嗎
  - 有反差嗎
  - 畫面有記憶點嗎
  - 其他人做也有機會跑嗎
- 列出外文 Shorts 參考時，一定要同時提供：
  - 原文標題
  - 中文翻譯
  - 影片連結
  - 觀看數快照
  - 發片時間
  - 所屬主題簇
  - 為什麼值得抄
  - 這題是不是可複製熱門
- 如果使用者指定某支參考 Shorts，除了片本身，也要去看那支片下面 `高讚留言` 在討論什麼。
- 高讚留言不是拿來照抄，也不是要硬寫成 `留言說`、`有人說`。
- 正確做法是把留言裡反覆出現的觀察、笑點、誤讀點、第二層解讀，整理成文稿裡更自然的比較、補刀或結論。
- 預設讓使用者先從候選題型裡自己挑一個，再進到全文撰寫。

## Hook 規則

- 第一行是 Shorts 最重要的句子，要當成獨立任務先做。
- 預設先研究 `日文 + 英文` 爆紅 Shorts 的前 1 句，再決定中文開頭，不要只靠自己硬想。
- 如果手上已有固定參考片，先拆那支片的 `前 1 句怎麼鉤人`，再重寫成中文。
- 如果使用者指定不要看中文圈，就不要拿中文爆款當第一句參考。
- 第一行優先學的是 `勾子結構`，不是逐字翻譯。
- 常見可抄的結構只有這幾種：
  - `你有注意到嗎？`
  - `怎麼沒人告訴我...`
  - `原來...其實...`
  - `最離譜的是...`
  - `你以為...其實...`

## 跟長影片的差別

- 長影片可以拆章節、拆段、補機制。
- Shorts 不行。
- 長影片是在帶玩家理解完整做法。
- Shorts 是在用一個最強角度，讓觀眾立刻被打中，再把文字拆成剪輯可用的版型層級。

## 禁止行為

- 不要寫成長影片濃縮版。
- 不要加章節名。
- 不要寫長影片式章節名；如果真的需要切節奏，用很短的 `步驟提示卡`。
- 不要塞太多背景知識。
- 不要前面鋪太長才講重點。
- 不要只靠情緒，沒有資訊支撐。
- 不要失去 `玫玫物語` 原本的判斷感與帶路感。
- 不要寫成一般解說型旁白。
- 不要忘記使用者指定的參考 Shorts 節奏。
- 不要把參考 Shorts 的留言生硬塞進稿子裡。
- 不要一直用 `不是...而是...` 去講 punch，尤其是有趣點、離譜點、穿幫點。
- 不要讓 `HOOK 爆字`、`前言穩字`、`步驟提示卡`、`教學穩字` 混成一坨。
- 不要把 `教學穩字` 硬寫成兩行量。

## 固定輸出格式

```md
# Shorts Package

## Hook Title
## Hook Burst Text
## Template Marked Script
```

## 一句話版交接

Shorts 的任務不是完整講清楚，而是抓一個最值得講的點，用 `玫玫物語` 的口氣更快、更狠地打出去，讓觀眾第一句就停下來，最後一句被打中。
