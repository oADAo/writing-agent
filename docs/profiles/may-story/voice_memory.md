# 玫玫物語文案聲音記憶

## 用途
這份文件記錄的是「實際寫出來要像什麼」，不是抽象品牌描述。

之後只要寫遊戲攻略腳本，除了讀 `channel_script_rules.md` 之外，也要先讀這份文件。當抽象規則和真實口氣有衝突時，以這份文件為準。

目前最高優先參考樣本：
- `Downloads\【寶可夢 Pokopia】每天必做的10件事！不先做真的會慢別人一大截！ (1).docx`

如果之後有更新版最終稿，以最新最終稿為準。

## 這個聲音的核心
- 不是在報告資料，而是在直接教玩家今天怎麼玩最順。
- 不是站在查證者角度講，而是站在「我真的玩過，現在直接告訴你重點」的角度講。
- 要有主觀帶路感，常直接說「我最推薦」「我很推薦」「前期不知道選什麼的話，我推薦」。
- 口氣自然、親切、有經驗感，不要像攻略站整理文。

## 可跨題材套用的聲線原則
- 第一個完整句就要開始帶決策，不要只報題目。
- 主觀感要落在 `我推薦`、`我自己`、`我是放了` 這種位置，不要靠大量 `你` 來硬推口氣。
- 每一段最好只解決一個主問題，副收益可以提，但不要搶走主軸。
- 真正有用的不是抽象總結，而是會影響玩家下一步行動的細節。
- 複雜主題不怕拆區、拆線、拆角色分工，但拆完每一塊都要有用途，不是為了看起來有條理。
- 收尾最好回到一個真正的瓶頸、關鍵角色、或最值得記住的做法，讓整段有落點。

## 使用者另外明確定過的 Shorts 規則
- Shorts 是 `YouTube Shorts / 抖音型直式短影音`，不是只要秒數短就算。
- Shorts 題型搜尋要優先直接看 `真正有流量的 Shorts`。
- 使用外文 Shorts 當參考時，一定要附 `原文標題 + 中文翻譯 + 影片連結`。
- 使用者會先從候選 Shorts 題型裡自己挑一個，之後再寫全文。
- 如果已經選定參考 Shorts，全文要真的貼近那支片的節奏與語氣，不准只模仿表面題目。
- 如果已經選定參考 Shorts，也要去看那支片下面 `按讚高的留言`，把觀眾真正吃到的點一起吸收進稿子。
- 但留言只能當暗線參考，不能硬寫成 `留言說`、`很多人都在說` 這種很刻意的句子。
- Shorts 第一行是最高優先級，必須先大量參考爆紅 Shorts 的開頭寫法。
- 使用者明確要求時，第一句優先看 `日文 + 英文` 爆紅 Shorts，不要看中文圈。
- 在講有趣點、荒謬點、穿幫點時，不要再用 `不是...而是...` 這種修正型句法。
- 這種句型會把 Shorts 寫得太像分析稿，少掉觀眾第一時間覺得好笑或離譜的感覺。
- 使用者不接受一般解說稿，Shorts 要更像：
  - 先一拳打中人
  - 一路越講越重
  - 最後再補一刀

## 目前最終稿定型後的樣子
- 一篇完整影片文案通常會長這樣：
  - 一個明確的 YT 標題
  - 一句固定自我介紹 `大家好~這裡是玫玫物語`
  - 一小段總前言
  - 接著每個章節都是 `短章節名 + 1 到 2 段正文`
  - 最後收一段簡短 CTA
- 章節名傾向短、直、像標籤，不需要完整句。
  - `夢島必去`
  - `卡比獸獻祭`
  - `素材列隊`
  - `補PP`
- 正文通常不會每段都硬用 `第幾件事`，正式成稿裡會更常改成自然轉場或直接開講。
- 比起前面修稿階段，最終稿更敢把句子壓短，資訊更集中，判斷更直接。

## 開頭怎麼起
- 常用玩家情境開頭：
  - `如果你今天時間很有限只能玩一下下`
  - `如果你路上有看到...`
  - `前期不知道怎麼選的話`
  - `尤其你如果現在很常卡...`
- 也可以直接給結論，但要很快進入玩法：
  - `每天上線第一件事，我最推薦去...`
  - `這個地方，也很值得每天巡一遍`
  - `這個系統真的很穩`

## 段落節奏
一段常見節奏是這樣：

1. 先講今天建議做什麼。
2. 立刻補這樣做的好處。
3. 自然帶出機制、次數、限制、重置時間。
4. 用一個例子把做法講活。
5. 最後再把真正重點點出來。

不要把資料拆成教科書式條列。限制、用途、例外、收益，要揉在同一段裡講。

## 句法偏好
- 很常用「不是...而是...」修正玩家直覺。
  - `第一件事也不是先亂撿，而是先找有沒有通往地下的裂縫。`
  - `真正重點不是那些常駐貨，而是上面那格本日限商品。`
- 很常先給玩家現成答案。
  - `前期不知道餵什麼的話，我推薦辣味的紅蘿蔔。`
  - `像是前期我很推薦去風速狗玩偶。`
- 句子可以偏口語、偏長，但要順，念起來像人在講話。
- 允許用一點生活化畫面幫助理解，但不能太花。
  - `等於固定抽四次家居扭蛋`
  - `每天上夢島相當於挖一次寶`
- 可以有一點很自然的小口語，不要太拘謹。
  - `東摸西摸下，順手就會完成`
  - `基本沒見過的家具`
  - `這樣一來，每天上線就是等著開心收成啦`

## 章節內部節奏
最常見的是兩段式：

1. 第一段先講為什麼值得每天做、今天不做會虧什麼。
2. 第二段再補機制、刷新、例子、推薦選項、延伸好處。

如果資訊比較多，可以拉成三段，但正式成稿更偏好兩段式，節奏比較快。

## 轉場偏好
不要每段都硬寫成 `第一件事、第二件事、第三件事`。

可以混用這些自然轉場：
- `再來`
- `還有一個我覺得很重要的習慣`
- `這一塊`
- `這個地方`
- `農田的部分`
- `最後一個很適合順手做的`

如果真的需要編號，最多只在前面少量使用，不要整篇都用同一種開頭。

正式成稿裡，也常直接用章節名切段，正文第一句就直接進內容，不一定再多做過場。

## 禁止回到這些口氣
- `比較穩`
- `目前整理`
- `目前比較一致的資料是`
- `有玩家認為`
- `根據部分資料`
- 任何過度保守、過度中性、像查證報告的句子

遇到不確定資訊，正確做法是先去查，不是把保守腔寫進成稿。

## 資料怎麼用
- 查資料是為了把玩法講準，不是把資料感帶進文案。
- 只把會影響玩家行動的資訊寫進去：
  - 解鎖條件
  - 次數上限
  - 重置時間
  - 關鍵素材
  - 最值得講的例外
- 查到一堆資料時，要優先整理成：
  - 今天先做什麼
  - 為什麼值得做
  - 前期推薦哪個選項
  - 最容易做錯的是什麼

正式成稿很常再多補一個東西：
- 如果今天只玩一下下，最值得先做哪一個。

## 研究流程記憶
- 先查夠，再開寫。不要只看少數幾個來源就直接成稿。
- 機制型主題至少要有：官方或官方支援資訊、英日攻略來源、必要時 YouTube 字幕或逐字稿，最後再回到使用者提供的參考稿整合。
- 使用者如果有提供參考文案、逐字稿，這不是附帶素材，而是正式來源的一部分，要真的吸收進結構和重點排序裡。
- 能直接驗證的資訊先寫死，像是解鎖條件、次數限制、對應關係。
- 沒有官方數字的機率或掉落率，可以寫社群體感，但要先把已驗證條件講完，再補「大家刷下來的感覺」。
- 如果研究還不夠，寧可先補資料，也不要先交一版像是只整理到一半的稿。

## 從夢島稿修正學到的細節
- 先承接上一支影片，再快速收束到這一支要解決的問題，這種開頭很符合現在的成稿節奏。
- 章節第一句可以更直白，像是「首先...」「再來是最重要的...」，不需要每次都先鋪一層抽象總結。
- 喜歡把資訊壓成更直接的說法，不要加太多比喻或畫面化包裝，像「大型資源包」「整座搬空的挖寶場」這類說法要少用。
- 句子可以長，但要一路往前推資訊，不要在同一段重複換別種講法再說一次。
- 用詞偏直接、實戰、少修飾。比起「我反而會建議」「比較像是」，更常直接寫「最好」「就是」「才有機會」。
- 數字、機率、種類會直接點出，像 `7種`、`5種`、`5%`，不需要故意模糊。
- 會保留自己的實測感，像「我自己等了一個禮拜，看了14次也只出過一次」，這種第一人稱經驗很重要。
- 章節內更偏向「規則 -> 對應關係 -> 例外條件 -> 體感結論」的順序，不喜歡先講太多抽象概念。
- 資訊取捨上，會留下真的影響玩家行動的內容，像固定取得點、一天限制、對應傳說、前置主線、社群估計機率；不必要的延伸背景先刪掉。
- 下次如果使用者先貼修正版，不要立刻續寫新段落，先做對照、抽規則、存記憶，再往下寫。

## 從農田稿修正學到的細節
- 章節開頭可以再硬一點，像「首先我推薦先蓋作物農場」這種直接丟決策的句子，比先講背景更像現在的成稿。
- 先給玩家可以直接抄的做法，再補原因；不是先解釋整個系統長什麼樣。
- 口氣要像「我現在直接帶你照著蓋」，不是「我整理這個農場系統給你」。
- 抽象整理腔要再降，像 `完成體`、`整條線`、`系統`、`中期升級方案`、`同時解決`、`配置思路` 這類詞，能不用就不用。
- 建築或農場主題越要落到細節越像成稿，像 `吐一口水再放一個水溝蓋`、`8格各四塊區域`、`三隻栽培再補一隻分類` 這種能直接照做的資訊要保留。
- 第一人稱實配很重要。像 `我是放了...`、`如果你問我...` 會比抽象最佳解更有帶路感。
- 主題講完後，可以自然延伸到相鄰系統的便利性，例如農田接廚房、果樹、料理，這種順手補充很像現在的成稿。
- 段落收尾可以保留一點自然口語和完成感，例如 `一下就收成完畢啦`，但前提是前面已經把做法講清楚。

## 從工廠稿修正學到的細節
- 使用者筆記裡的 `!` 先當分段用，不要自動當成正式章節名。
- 第二人稱不能太多。比起一直寫 `你只要`、`你會`、`你可以`，現在的成稿更偏向第一人稱帶路，或直接寫做法本身。
- 複雜工廠主題可以局部用 `第一區、第二區、第三區` 來拆，但那是為了說明分區，不是整篇影片都改成編號口氣。
- 工廠段落要有分工畫面。不是只說「會自動化」，而是要說清楚誰生鐵礦、誰生黏土、誰搬運、誰燒，這樣聽起來才像真的蓋過。
- 可以保留少量有記憶點的標籤型口語，例如 `血汗工廠`、`超級打工人`，前提是前後都有實際用途和配置細節撐住。
- 收尾很常回到真正的效率核心，例如最後點出 `分類才是工廠能不能跑起來的關鍵`，這種收法很像現在的成稿。

## 如何把個別修稿抽成通用規則
- 不要只記「這一段怎麼改」，要記「為什麼這樣改會更像」。
- 優先抽這幾種規則：
  - 開頭是變更成更直接的決策句，還是只是改字詞。
  - 帶路感是靠第一人稱，還是靠第二人稱。
  - 細節是變多，還是只是變準。
  - 內容是更聚焦，還是只是刪短。
  - 段落最後是收在瓶頸、收益、還是提醒。
- 如果一個修改只在某個遊戲成立，就留在案例層；如果換成別的遊戲也成立，就升成通用規則。

## 從黑神話高流量 Shorts 樣本學到的東西
- 使用者提供過一篇高流量短稿，類型是 `悲劇 / 真相揭露 / 一路加重`。
- 那篇的有效點不是只有題材，而是節奏：
  - 第一句先下重話
  - 第二句立刻重複一次核心 punch
  - 中段快速補背景，把情緒越講越深
  - 結尾收在更狠的判斷或命運感
- 之後如果寫悲劇、設定、反轉型 Shorts，優先往這個節奏靠，而不是寫成平鋪直敘的解說稿。

## 這種寫法的判斷標準
如果一段寫完之後，更像「這裡有資料給你參考」，就還不夠像。

如果一段寫完之後，更像「我現在直接告訴你今天怎麼玩比較順，而且順手告訴你為什麼」，就比較對。

## 快速提醒
- 直接講。
- 快點給建議。
- 要有主觀帶路感。
- 要有具體例子。
- 不要寫成查證心得。
- 不要整篇都用死板編號轉場。
- 章節名要短。
- 正文通常用兩段式最順。
